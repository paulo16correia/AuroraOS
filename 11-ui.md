# Aurora OS — RFC 11: User Interface and Human Operations

**State:** Normative · **Depends on:** RFC 10

## Objective

Define an interface that makes Aurora controllable, understandable and fixable. The UI is not just a chat window; It is the command post for memory, objectives, permissions, approvals, integrations and auditing.

## Responsibilities

- Present conversations and work states without hiding uncertainty or pending actions.
- Allows you to manage memories, goals, tools, profile and personal data.
- Provide deliberate approval for impact actions and background investigations.

## Mandatory rules

1. The UI MUST clearly distinguish generated content, draft, proposed action, ongoing action and completed action.
2. Approval MUST show recipient/purpose, sensitive data involved, validity and rejection option; never use ambiguous buttons.
3. The user MUST be able to see the origin, trust, and lifecycle of a memory and perform correction or deletion.
4. Sensitive data MUST be hidden by default and revealed only by explicit, temporary and authorized gesture.
5. The interface MUST NOT suggest that Aurora is running in the background when there is no verifiable active task.

## Limit and error cases

- **Outdated status:** indicate update time and reload before approving or changing.
- **Two windows approve:** the first valid decision consumes approval; the second gets conflict, not a duplicate execution.
- **Accessibility:** all critical actions must be usable via keyboard, screen reader and without relying solely on color.
- **Degraded API:** show read-only mode or unavailability; do not accept actions that cannot be confirmed by the server.

## Justification

On a system with memory and tools, transparency is not a secondary screen. An operational UI reduces human error and makes user control real, not just declared.

## Future expansions
