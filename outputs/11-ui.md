# Aurora OS — RFC 11: Interface de utilizador e operações humanas

**Estado:** Normativo · **Depende de:** RFC 10

## Objetivo

Definir uma interface que torne a Aurora controlável, compreensível e corrigível. A UI não é apenas uma janela de chat; é o posto de comando para memória, objetivos, permissões, aprovações, integrações e auditoria.

## Responsabilidades

- Apresentar conversas e estados de trabalho sem esconder incerteza ou ações pendentes.
- Permitir gerir memórias, objetivos, ferramentas, perfil e dados pessoais.
- Oferecer aprovação deliberada para ações de impacto e investigação de histórico.

## Arquitetura e diagrama

```text
Web/Desktop/Móvel
        │ HTTPS + sessão autenticada
        ▼
Cliente de UI ──→ API RFC 10 ──→ serviços de domínio
        │                │
        ▼                ▼
estado local mínimo   stream de atualizações

Painel
├─ Conversa e trabalhos
├─ Objetivos e tarefas
├─ Memória e grafo
├─ Aprovações e ferramentas
└─ Auditoria, saúde e definições
```

## Estruturas de dados

```text
UiViewState
  route, selected_subject_ref, filters, cursor, locale
  freshness_at, permissions[], notification_preferences

ApprovalCard
  approval_id, action_summary, target_summary, risk
  data_disclosure_summary, expires_at, approve_label, reject_label
  immutable_scope_hash

ActivityItem
  id, occurred_at, category: MESSAGE|TASK|TOOL|SECURITY|SYSTEM
  status, title, summary, subject_ref, evidence_link
```

## Interfaces e fluxos

```text
Ui.load(route, cursor) -> UiViewState + data
Ui.approve(approval_id, scope_hash) -> Approval
Ui.correctMemory(memory_id, revision) -> MemoryRevision
Ui.cancelWork(work_item_id) -> WorkItem
```

```text
ação proposta → cartão de aprovação → utilizador revê âmbito
       │                                     │
       └──────── rejeita ────────────────────┤
                                             ▼
                                   aprova hash exato → API
```

## Regras obrigatórias

1. A UI DEVE distinguir claramente conteúdo gerado, rascunho, ação proposta, ação em curso e ação concluída.
2. Aprovação DEVE mostrar destinatário/efeito, dados sensíveis envolvidos, validade e opção de rejeição; nunca usar botões ambíguos.
3. O utilizador DEVE poder ver origem, confiança e ciclo de vida de uma memória e efetuar correção ou eliminação.
4. Dados sensíveis DEVEM estar ocultos por defeito e revelar-se apenas por gesto explícito, temporário e autorizado.
5. A interface NÃO DEVE sugerir que a Aurora está a agir em segundo plano quando não existe tarefa ativa verificável.

## Casos limite e erro

- **Estado desatualizado:** indicar hora de atualização e recarregar antes de aprovar ou alterar.
- **Duas janelas aprovam:** a primeira decisão válida consome a aprovação; a segunda recebe conflito, não uma execução duplicada.
- **Acessibilidade:** todas as ações críticas devem ser utilizáveis por teclado, leitor de ecrã e sem depender só de cor.
- **API degradada:** mostrar modo só de leitura ou indisponibilidade; não aceitar ações que não possam ser confirmadas pelo servidor.

## Justificação

Num sistema com memória e ferramentas, transparência não é um ecrã secundário. Uma UI operacional reduz erro humano e torna o controlo do utilizador real, não apenas declarado.

## Expansões futuras

Visualizador de grafo, linha temporal de decisões, modo de voz, clientes nativos, notificações com regras de silêncio e espaço de trabalho para equipas.
