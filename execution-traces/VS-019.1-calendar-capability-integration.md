# VS-019.1 — Capability Calendar Integration

## Scope

`CALENDAR_CREATE_EVENT` is a distinct external capability. The Kernel recognizes a meeting creation request, creates a `CapabilityRequest`, evaluates the policy, materializes a `CalendarEventDraft`, prepares the execution and persists a `ApprovalRequest` before touching the Google API.

```text
Structured message
        ↓
CapabilityRequest(CALENDAR_CREATE_EVENT)
        ↓
Policy(REQUIRED)
        ↓
CalendarEventDraft
        ↓
ExecutionPreparation(PENDING_APPROVAL)
        ↓
ApprovalRequest(PENDING)
        ↓
Approval(APPROVED)
        ↓
GoogleCalendarProvider.create_event
        ↓
CapabilityResult(SUCCESS | FAILED)
```

## Initial entry contract

To avoid temporal ambiguity at this stage, the order uses explicit ISO 8601 dates:

```text
Set up a meeting | Title: Review | Start: 2026-07-20T15:00:00+01:00 | End: 2026-07-20T16:00:00+01:00 | Guests: joao@example.com
````Title`, `Start` and `End` are required; `Guests` and `Time zone` are optional. An invalid preparation is blocked before requesting approval.

## Guarantees

- No events are created before `Approval(APPROVED)`.
- `ExecutionRecord` is persisted before the external call.
- The same execution is not automatically repeated after failure or restart.
- Aurora idempotency key is sent as Google event ID and as private property of the event.
- The availability query (`freeBusy`) remains a future increment: there is still no reading capability linked to the Kernel.
