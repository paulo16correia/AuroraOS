# Aurora OS — Execution Trace Specification: VS-007.1 Capability Pipeline Alignment

**Estado:** Autorizado por ADR-007  
**Caso de utilização:** Entity com Task histórica → novo pedido de email → CapabilityRequest obrigatória

## Objetivo

Garantir que um pedido de capacidade nunca é confundido com ausência de necessidade. A capacidade pode estar indisponível, mas a intenção continua a requerer avaliação explícita.

## Fluxo normativo

```text
Task(COMPLETED) persistida de ciclo anterior

User: "Envia um email por mim"
  → TaskLoaded (auditoria)
  → Capability source = Task(COMPLETED) persistida
  → CapabilityRequest(EMAIL_SEND, UNAVAILABLE)
  → CAPABILITY_ASSESSMENT(UNAVAILABLE)
  → Thought(reason = request validada)
  → Decision(RESPOND)
  → CAPABILITY(UNAVAILABLE)
  → Response

GoalEvaluation
  → não recebe a Task histórica
  → não duplica progresso
```

## Regras obrigatórias

1. Intenção de email DEVE produzir ou carregar `CapabilityRequest` antes de Thought.
2. `CAPABILITY(NOT_REQUIRED)` é proibido quando existe uma intenção reconhecida que requer capability.
3. Só Tasks concluídas no ciclo atual podem alterar `GoalEvaluation`.
4. Uma Task histórica pode fundamentar uma request, mas não pode ser reexecutada.
5. Nenhuma Action, ToolCall, provider ou efeito externo é introduzido.

## Critérios de aceitação

1. Após “Como podes ajudar-me?”, um pedido posterior “Envia um email por mim” gera `CapabilityRequest(EMAIL_SEND, UNAVAILABLE)`.
2. O trace contém `TASK(LOADED)`, `CAPABILITY_REQUEST`, `CAPABILITY_ASSESSMENT` e `CAPABILITY(UNAVAILABLE)`.
3. O Thought e a resposta referem a request validada, não uma ausência de request.
4. O Goal progress não aumenta no segundo ciclo por reutilizar a Task histórica.
5. A suite existente permanece verde.
