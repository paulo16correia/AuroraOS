# Aurora OS — Execution Trace Specification: VS-007.1 Capability Pipeline Alignment

**Status:** Authorized by ADR-007
**Use case:** Entity with historical Task → new email request → Mandatory CapabilityRequest

## Objective

Ensure that a request for capacity is never confused with an absence of need. The capability may be unavailable, but the intent continues to require explicit assessment.

## Normative flow

```text
Task(COMPLETED) persistida de ciclo anterior

User: "Send an email for me"
  → TaskLoaded (auditoria)
  → Capability source = Task(COMPLETED) persistida
  → CapabilityRequest(EMAIL_SEND, UNAVAILABLE)
  → CAPABILITY_ASSESSMENT(UNAVAILABLE)
  → Thought(reason = request validada)
  → Decision(RESPOND)
  → CAPABILITY(UNAVAILABLE)
  → Response

GoalEvaluation
→ does not receive the historical Task
→ does not duplicate progress
```

## Mandatory rules

1. Email Intent MUST produce or load `CapabilityRequest` before Thought.
2. `CAPABILITY(NOT_REQUIRED)` is prohibited when there is a recognized intent that requires capability.
3. Only Tasks completed in the current cycle can change `GoalEvaluation`.
4. A historical Task can support a request, but cannot be reexecuted.
5. No Actions, ToolCalls, providers or external effects are introduced.

## Acceptance criteria

1. After “How can you help me?”, a subsequent request “Send an email for me” generates `CapabilityRequest(EMAIL_SEND, UNAVAILABLE)`.
2. The trace contains `TASK(LOADED)`, `CAPABILITY_REQUEST`, `CAPABILITY_ASSESSMENT` and `CAPABILITY(UNAVAILABLE)`.
3. Thought and response refer to the validated request, not an absence of request.
4. Goal progress does not increase in the second cycle due to reusing the historical Task.
5. The existing suite remains green.
